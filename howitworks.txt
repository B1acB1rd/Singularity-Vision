SINGULARITY VISION
Detailed Technical Feature & System Specification
Engineering Reference Document  |  Confidential
0. Global Architecture
Singularity Vision is a hybrid desktop system designed for professional computer vision workflows. The architecture is structured to ensure that the user interface remains highly responsive while all heavy computation is managed by isolated backend services.
Core Technology Stack
Layer	Technology
Frontend	Electron (React / Vue)
Backend Orchestrator	Python (FastAPI / gRPC / IPC)
Execution Engines	Modular Python services
Local Storage	File system + SQLite / DuckDB
Remote Services	Optional REST / gRPC APIs

Core Design Principle
The UI never performs heavy computation.
The Python backend never blocks the UI thread.
Heavy tasks are isolated, optional, and offloadable.


1. Project & Workspace System
Each project is a self-contained directory with a strict schema, designed to ensure portability, reproducibility, and minimal memory usage.
A. Project Directory Structure
Every project follows a standardised directory layout on disk:
project_root/
├─ project.json               # Core metadata & configuration
├─ datasets/                  # Raw + processed datasets
│    ├─ images/
│    ├─ videos/
│    ├─ geo/
│    └─ tiled/
├─ annotations/               # All annotation files
├─ models/                    # Saved / imported models
│    ├─ onnx/
│    ├─ pytorch/
│    └─ tensorflow/
├─ experiments/               # Training & evaluation experiments
│    ├─ experiment_<uuid>/
│    │    ├─ config.json
│    │    ├─ checkpoints/
│    │    ├─ logs/
│    │    └─ metrics.json
├─ spatial/                   # Spatial vision / mapping assets
├─ outputs/                   # Inference outputs, visualizations
└─ logs/                      # Application & engine logs

B. project.json — Core Metadata
The project.json file is the single source of truth for a project. It drives the UI, engines, and all workflows.
{ "project_id": "uuid", "name": "Mining Site Monitoring",
  "industry_profile": "mining", "task_types": ["detection", "mapping"],
  "execution_mode": "hybrid", "security_level": "offline_allowed",
  "engine_versions": { "dataset_engine": "v1.0.2", ... } }

C. Key Project Behaviors
Lazy Loading
Only project.json is loaded on open. Datasets, models, experiments, and outputs are loaded on demand to minimise memory footprint and improve startup time.
Versioning
Every dataset change creates a hash, a diff record, and a reversible snapshot. Silent overwrites are not permitted. Each training run is stored as an immutable experiment snapshot containing its configuration, checkpoints, logs, and metrics.
Export / Import
Full project exports include all assets and are bundled as a zip with a manifest.json that records SHA256 hashes of every file for integrity verification.
Project Isolation
Projects are fully self-contained. Engines are aware of project paths and never write outside the project folder by default, preventing accidental cross-project contamination.
Security & Access Control
Project folders can be encrypted for sensitive industries. Execution mode can enforce offline-only behaviour or allow cloud engines for heavy tasks. Audit logs record all inference runs, experiment training events, and remote vs. local execution.

2. Dataset & Data Management Engine
The Dataset Engine runs as a background Python service. Heavy operations such as tiling, parsing, and indexing are offloaded to worker threads or processes. The Electron frontend communicates via async IPC / gRPC / REST.
A. Supported Input Types
Input Type	Libraries	Notes
Images	Pillow, OpenCV	Copied or referenced into datasets/images/
Video	OpenCV, FFmpeg	Frame metadata stored separately
Satellite Imagery	Rasterio, GDAL	Large images automatically tiled on ingest
GeoTIFF	GDAL, Rasterio	Parsed and linked to spatial metadata
CSV / Tabular Spatial	Pandas, GeoPandas	Parsed into spatial index

B. Data Ingestion Pipeline
The ingestion pipeline follows five sequential stages: Validation → Metadata Extraction → Optional Tiling → Indexing → Dataset Registration.
•Validation: File format checks, corrupted image detection, CSV / GeoTIFF header validation.
•Metadata Extraction: Each asset receives a unique ID, file path, type, dimensions, timestamp, optional geo-bounds, and source tag. All metadata is stored in SQLite / DuckDB for fast querying without loading raw files.
•Tiling: Images exceeding the configured threshold (e.g., 2048x2048) are split into tiles. Each tile stores its original coordinates, pixel coordinates, and index, enabling memory-efficient processing of large satellite imagery.
•Indexing: Assets are registered in the dataset_index table with file path, type, timestamp, spatial bounds, and class labels. An optional R-tree provides geo-spatial query capability.

C. Dataset Versioning & Snapshots
Each ingestion or update generates a SHA256 hash per asset and compares with the previous version to identify changes. A diff record and a version ID are assigned. Snapshots are read-only references, allowing full rollback to any prior state.
D. Geo-Spatial Awareness
Assets with geographic coordinates are registered in an R-tree index enabling bounding-box queries (all assets within a lat/lon region), fast map rendering, and CV result overlays on spatial maps. Each frame or tile stores lat_min, lat_max, lon_min, lon_max, optional altitude, and optional timestamp for temporal analysis.

3. Annotation & Labeling System
The Annotation Engine runs as a Python backend service. The Electron frontend sends geometry, label, and metadata via IPC / REST / gRPC. Annotations are stored in a central registry and are never baked into source images.
A. Annotation Storage Model
Core principles: annotations are decoupled from images and videos; raw source files are never modified; annotations are stored as metadata JSON referencing an image_id or frame_id.
Supported geometry types:
•Bounding boxes (x, y, width, height)
•Polygons (list of points)
•Polylines (roads, rivers, infrastructure)
•Segmentation masks (reference mask file)
•Temporal segments (start/end frame for video)
•Keypoints / pose landmarks (Phase 2)

B. Annotation UI to Backend Flow
The user draws an annotation in the Electron UI, selecting a label from the project-defined label set and optionally assigning a confidence value. The frontend sends the geometry, image_id, frame_index, and label to the backend via IPC or gRPC. The backend validates geometry bounds, label existence, and geo-bounds, then stores the annotation in the SQLite registry and updates dataset statistics.
C. Geo-Aware Annotations
If an asset has geo-metadata, pixel coordinates are automatically converted to geospatial coordinates using GDAL / Rasterio. Both the pixel geometry and the lat/lon geometry are stored. This enables map rendering in the Electron UI, spatial filtering, integration with the Spatial Vision Lab, and real-world distance and area calculations.
D. Annotation Versioning & History
Every annotation tracks version metadata. Edits copy the previous version and increment the counter. An audit log records the user, timestamp, and action type (create / update / delete), enabling full rollback and experiment reproducibility.
E. Supported Export Formats
Format	Use Case
YOLO (.txt)	Model training pipelines
COCO (.json)	Standard detection / segmentation format
Pascal VOC (.xml)	Legacy and cross-tool compatibility
GeoJSON	GIS tools and spatial analysis


4. Computer Vision Tasks
The Vision Engine is a Python service handling training, inference, and evaluation. It supports local CPU/GPU execution and optional remote acceleration. All CV tasks implement a common interface to ensure consistent, modular behaviour across the platform.
A. Task Abstraction Interface
class VisionTask:
    def load_model(self):   # Load ONNX / PyTorch / TF model
    def preprocess(self, data):  # Resize, normalise, tile
    def infer(self, preprocessed_data):  # Run inference
    def postprocess(self, raw_outputs):  # Convert to bbox / mask / geo

This interface enables model swapping, local or remote execution, ONNX standardisation, and consistent behaviour across all task types.
B. Change Detection
Change detection identifies differences between two or more temporally-separated images, with optional geo-awareness. It is critical for mapping, defence, mining, and industrial monitoring applications.
Pipeline
•Step 1 — Preprocessing: Image alignment using keypoint-based registration (SIFT / ORB / SuperPoint) or geo-coordinates. Resizing, normalisation, and tiling of large images.
•Step 2 — Comparison: Pixel or feature differencing for simple use cases; or a learned CNN / UNet / Siamese network that accepts stacked image pairs and outputs a per-pixel probability map of change.
•Step 3 — Postprocessing: Thresholding the probability map to a binary mask, morphological noise removal, and calculation of change statistics. Geo-linked overlays are generated as GeoJSON polygons and GeoTIFF raster masks.

Engine Integration
Engine	Interaction
Dataset Engine	Pulls tiles/images; provides geo metadata
Spatial Engine	Converts pixel change mask to geo-coordinates; overlays on map
Annotation Engine	Optionally auto-annotates changes as bounding boxes or polygons
3D Engine	Projects change masks into 3D models for volumetric analysis
UI / Electron	Displays interactive map/video overlays; supports mask toggling


5. Model Hub & Model Management
The Model Manager is a Python backend service providing a centralised, versioned, and extensible model management system. Models are ONNX-first for cross-framework compatibility.
A. Model Registry Structure
Each model is stored as a self-contained directory:
models/<model_id>/
├─ model.onnx         # Main model file
├─ config.json        # Architecture, input/output specification
├─ metrics.json       # Accuracy, mAP, inference speed
└─ runtime.json       # Device requirements, supported tasks

B. Hugging Face / Remote Integration
•Model metadata is queried via API (name, task type, weights format, licence).
•Weights are downloaded on demand and converted to ONNX if required.
•Models are cached locally. No permanent online dependency exists.
•Users can continue offline if a model is already cached.
•Enterprise deployments can use internal model repositories in place of Hugging Face.

C. Model Search & Filtering
The frontend provides filtering by task type (detection, segmentation, OCR, change detection), device compatibility (CPU / GPU), industry preset (mining, defence, sports), and minimum metrics threshold (e.g., mAP >= 0.85). The backend queries the registry and returns model metadata ready for selection and execution.

6. Training & Evaluation Engine
The Training Engine runs as a separate Python process fully decoupled from the Electron UI. The UI interacts asynchronously via IPC / WebSocket / REST. Both local and remote/cloud training use an identical config structure to prevent logic duplication.
A. Training Flow
Dataset → Config → Trainer → Checkpoints → Metrics → Exportable Model
B. Config Object
{ "model_id": "yolov8_mining_v1", "task": "detection",
  "dataset_id": "dataset_001", "batch_size": 8, "epochs": 50,
  "optimizer": "Adam", "learning_rate": 0.001, "device": "gpu",
  "augmentation": ["flip", "rotate"], "checkpoint_interval": 5,
  "metrics": ["mAP", "accuracy", "loss"] }

C. Key Engine Behaviours
•Trainer: Runs as an isolated Python process to protect UI responsiveness. Supports mixed precision (FP16), multi-GPU, and configurable early stopping.
•Checkpoints: Saved automatically at configurable intervals. Each checkpoint includes model weights, optimiser state, training step, and a config snapshot for full reproducibility.
•Metrics: Computed per epoch and per class. Saved as JSON and streamed to the UI dashboard for real-time loss/accuracy charts and confusion matrix visualisation.
•Export: After training, weights are converted to ONNX and packaged with config.json, metrics.json, and runtime.json, ready for the Vision Engine or Annotation Engine.

D. Local vs. Remote Training
The same config object is used for both local and remote execution. The only difference is the execution endpoint. Local training runs on the user's CPU or GPU and is fully offline-capable. Remote training sends the config to a cloud endpoint, which returns final weights, metrics, and checkpoints. The UI polls progress identically in both modes.

7. Inference & Testing Studio
The Inference Engine executes CV models against input data and returns structured visualisations. Inference never blocks the UI. All execution is asynchronous.
Execution Flow
Input → Preprocess → Model Runtime (ONNX) → Postprocess → Visualisation
For geo-referenced datasets, predictions are mapped back to spatial coordinates and rendered as toggleable vector layers in the Electron UI.

8. Spatial Vision Lab
The Spatial Engine is a core platform differentiator, enabling computer vision to operate natively on geographic data, maps, and temporally-sequenced imagery.
A. Spatial Data Model
All spatial assets share a common coordinate reference system (CRS) descriptor, bounds array, and resolution metadata. Supported formats include EPSG coordinate systems, GeoJSON, and GeoTIFF. All assets are registered in an R-tree spatial index.
B. Vision on Maps
Map layers are treated as tiled images. CV models run on individual tiles, and outputs are merged back into coherent spatial layers. This enables detection, segmentation, and change detection at geographic scale.
C. Temporal Mapping
The same spatial region captured at different timestamps can be compared. This supports before/after analysis and animated difference overlays, which are particularly valuable for environmental monitoring, mining, and construction projects.

9. 3D Digital Environment & Reconstruction
The 3D Engine converts image, video, and drone sequences into 3D digital environments and integrates CV task outputs into 3D space for measurement and analysis.
A. Component Overview
Component	Role
Local 3D Engine	Basic reconstruction via OpenMVG / COLMAP; CPU-based; small datasets
Online 3D Engine	Heavy-duty SfM / SLAM; GPU-accelerated; high-res datasets; optional
3D + CV Integration	Projects 2D detections into the reconstructed 3D environment
3D Data Store	Stores point clouds, meshes, textured environments, and metadata
Visualisation Module	Electron frontend with camera controls, overlays, and measurement tools

B. Local 3D (Basic)
Workflow: Feature extraction (SIFT / ORB) → Feature matching → Camera pose estimation → Sparse point cloud → Optional dense reconstruction via multi-view stereo → Texture mapping. Works on datasets of 10–100 images. CPU-based with no GPU dependency. Large images are automatically downsampled to avoid memory issues.
C. Online 3D (Advanced)
Only triggered when the user explicitly opts in for high-resolution or large-dataset reconstruction. Uses GPU-accelerated SfM / SLAM with drone GPS metadata to accelerate pose estimation. Supports a hybrid mode where the user previews a basic local reconstruction before submitting for high-resolution cloud refinement.
D. 3D + CV Integration
•Detection Projection: 2D bounding boxes are projected into 3D world space using camera intrinsics and extrinsics, producing 3D bounding boxes and mesh overlays.
•Measurements: Real-world distances between objects and volumes (e.g., mining pit volumes) are calculated using mesh or voxel models.
•Spatial Integration: The 3D environment is geo-referenced, allowing users to select a region on the 2D map and open the corresponding 3D scene.

E. Supported File Formats
Data Type	Formats
Point Clouds	.PLY, .LAS
Meshes	.OBJ, .FBX
Textures	.PNG, .JPG
Camera Poses	JSON (intrinsics, extrinsics)
3D CV Outputs	ONNX / JSON (3D bounding boxes, annotations)


10. Industry Profiles
Industry profiles are behavioural configuration files, not separate code. Core engines remain unchanged; profiles control which features are visible, allowed, or mandatory. This ensures regulatory compliance and faster adoption in sensitive industries.
A. Architecture & Storage
Each profile is a JSON / YAML file stored in config/industry_profiles/. On project creation, the selected profile is loaded and applied dynamically across the UI and all engines.
B. Feature Matrix
Feature	Defence	Mining	Health	Aviation	Sports
Cloud Training	Disabled	Enabled	Disabled	Enabled	Enabled
3D Reconstruction	Enabled	Enabled	Enabled	Enabled	Enabled
Volume Measurement	Disabled	Enabled	Disabled	Enabled	Disabled
OCR / Document Analysis	Disabled	Disabled	Enabled	Enabled	Enabled
Data Anonymisation	Mandatory	Optional	Mandatory	Optional	Optional

C. Runtime Enforcement
Both the UI and the engine layer enforce profile constraints at runtime. The UI hides or disables non-applicable controls. Engines perform pre-execution validation against the active profile constraints and raise errors before restricted actions are attempted.
D. Extensibility
New industry profiles require only a new JSON / YAML file. No code changes are needed. The platform is designed to support multi-profile projects for hybrid compliance requirements and user-defined corporate profiles.

11. Export & Interoperability
Exports are registry-driven. Each engine maintains a registry of outputs with metadata, file paths, and timestamps. The Export Engine reads these registries, converts formats, preserves metadata, and bundles files with a manifest.
A. Supported Export Types
Category	Formats	Notes
Models	ONNX, .pt, .pb, SavedModel	Includes config, metrics, runtime JSON
2D Datasets	COCO, YOLO, Pascal VOC, CSV	Includes annotations and class mappings
Geo-Aware Datasets	GeoJSON, GeoTIFF	Preserves lat/lon, bounds, timestamps
3D Reconstructions	PLY, OBJ, FBX	Point clouds, meshes, textures, camera poses
Reports / Metrics	CSV, JSON	Training metrics, change detection stats
Inference Scripts	.py	Auto-generated for trained models + preprocessing

B. Metadata Preservation
Every exported object automatically carries its full internal metadata: timestamps, spatial references, camera intrinsics and extrinsics for 3D outputs, dataset version hashes, and model hyperparameters. A manifest.json is generated for every export bundle, enabling full experiment reproducibility.
C. External Compatibility
•GIS Tools: QGIS, ArcGIS
•3D Tools: Unity, Unreal Engine, Blender
•ML Pipelines: PyTorch, TensorFlow, ONNX Runtime
•Cloud CV Platforms: Hugging Face, Roboflow (format-compliant)


12. Performance & Offloading
Every heavy task passes through a central Smart Task Orchestrator before execution. The orchestrator assesses hardware, dataset size, user plan, and security settings to determine whether execution should be local, remote, or hybrid.
A. Execution Decision Matrix
Dataset Size	GPU Available	Plan	Suggested Execution
Small	Yes / No	Any	Local
Medium	Yes	Free	Local
Medium	No	Free	Local (CPU — expect slower performance)
Large	Any	Premium	Hybrid / Remote
Extra Large	Any	Any	Remote

B. Local Resource Management
•Engine throttling limits thread and GPU usage to prevent UI freezing.
•Memory management uses lazy loading, streaming for large images/videos, and memory-mapped files for GeoTIFF and point clouds.
•A background task queue prioritises jobs based on user choice with async and parallel execution using Python multiprocessing or asyncio.

C. Hybrid Mode
Local preprocessing is combined with remote heavy computation. Example: tile satellite images locally, index datasets, run small model inferences; offload full model training or 3D reconstruction to a remote engine. Results are merged automatically with geo-referenced alignment.

13. Security & Scalability
Singularity Vision is offline-first by default. Cloud features are strictly opt-in and user data is never automatically uploaded.
A. Core Security Principles
Offline-first: No forced cloud upload. Full offline operation is always supported.
Encryption at rest: Optional AES-256 encryption for project files, datasets, annotations, and 3D reconstructions.
No forced cloud sync: Remote tasks always require explicit user confirmation.
Enterprise self-hostable: All core engines can be deployed on-premises using Docker / Kubernetes.

B. Data Security Measures
•File-level access control with per-project ACLs and multi-user collaboration controls for enterprise deployments.
•Transport security: all communication with remote engines uses HTTPS + TLS 1.3, with optional client certificates for enterprise.
•Project sandboxing: each project maintains separate dataset, annotation, and model registries. Engine processes are isolated with memory and file access restricted to the current project.
•Data anonymisation: optional stripping of PII, image region masking, and GPS coordinate obfuscation for health and defence deployments.
•Audit logs: append-only records of all data imports, training runs, exports, and remote offload events. Central logging is available for enterprise deployments.

C. Scalability
The platform is designed to run on a single laptop using lazy loading and tile-based processing, and to scale to enterprise GPU clusters and multi-node distributed execution using the same codebase. The Smart Task Orchestrator manages task distribution across GPUs and nodes. Optional remote offload scales automatically via a queued, multi-tenant GPU server.


FINAL ENGINEERING PRINCIPLE

Singularity Vision is not a tool with features. It is a system of coordinated engines with strict boundaries.

When engineers respect engine isolation, async execution, and local-first defaults,
this platform will scale reliably for years.
