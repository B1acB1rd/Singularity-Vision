PROJECT & WORKSPACE SYSTEM — DETAILED TECHNICAL SPECIFICATION

Goal: Each project is a self-contained, fully versioned workspace that can handle datasets, models, experiments, spatial data, and outputs, while minimizing memory usage and supporting offline/hybrid modes.

A. Project Directory Structure (Technical Schema)

Every project is a directory on disk, with strict subfolders and manifest files. This ensures portability, reproducibility, and easy syncing.

project_root/
 ├─ project.json               # Core metadata & configuration
 ├─ datasets/                  # Raw + processed datasets
 │    ├─ images/
 │    ├─ videos/
 │    ├─ geo/
 │    └─ tiled/
 ├─ annotations/               # All annotation files
 │    ├─ images/
 │    ├─ video_frames/
 │    └─ geo_annotations/
 ├─ models/                    # Saved / imported models
 │    ├─ onnx/
 │    ├─ pytorch/
 │    └─ tensorflow/
 ├─ experiments/               # Training & evaluation experiments
 │    ├─ experiment_<uuid>/
 │    │    ├─ config.json
 │    │    ├─ checkpoints/
 │    │    ├─ logs/
 │    │    └─ metrics.json
 ├─ spatial/                   # Spatial vision / mapping assets
 │    ├─ maps/
 │    ├─ geojson/
 │    └─ reconstructions/
 ├─ outputs/                   # Inference outputs, visualizations
 │    ├─ images/
 │    ├─ videos/
 │    └─ reports/
 └─ logs/                      # Application & engine logs


Notes:

Lazy-loading is critical: Datasets, models, and experiments are only loaded when accessed to save memory.

Portable: Entire project can be exported/imported as a zip with manifest.

Immutable Experiments: Once an experiment is finished, its snapshot (config + checkpoints + metrics) is preserved.

B. project.json — Core Metadata

This is the source of truth for a project.
It drives the UI, engines, and workflows.

Example:

{
  "project_id": "uuid",
  "name": "Mining Site Monitoring",
  "industry_profile": "mining",
  "task_types": ["detection", "mapping"],
  "execution_mode": "hybrid",        # local | hybrid | remote
  "security_level": "offline_allowed",
  "created_at": "2026-01-11T15:00:00Z",
  "last_opened": "2026-01-11T16:30:00Z",
  "engine_versions": {
      "dataset_engine": "v1.0.2",
      "vision_engine": "v2.1.0",
      "spatial_engine": "v1.5.0",
      "3d_engine": "v0.9.3"
  },
  "tags": ["urban", "satellite", "defense"],
  "user_settings": {
      "default_model": "yolo_v8",
      "default_output_format": "GeoJSON"
  }

Key Points:

execution_mode determines if tasks default to local/hybrid/remote.

security_level defines what is allowed: offline, online, cloud storage.

engine_versions ensures reproducibility: which engine version produced which result.

tags allow filtering/searching projects inside the platform.

user_settings store project-specific preferences (like default model, inference batch size, display units).

C. Project Behaviors
1. Lazy Loading

Only project.json is loaded on open.

Datasets, models, experiments, and outputs are loaded on demand.

Reduces memory footprint and improves startup time.

Example workflow:

User opens project → Electron UI displays metadata + dataset stats.

User clicks dataset → Python engine loads dataset (tiles, annotations) in memory.

User runs inference → only required model loaded into RAM.

2. Versioned Datasets & Models

Every dataset can have versions:

Original raw import

Augmented version

Preprocessed / tiled version

Models are versioned using:

uuid

Engine version

Training date

Purpose: Prevents overwriting important work and ensures reproducibility.

3. Experiments (Immutable Snapshots)

Each training run is an immutable snapshot.

Contains:

Configuration (config.json)

Model checkpoints

Logs

Metrics (metrics.json)

Experiment naming: experiment_<uuid> or user-friendly timestamp.

UI shows experiments in chronological order.

Experiments cannot overwrite each other, but can fork from previous snapshots.

4. Project Export / Import

Full export includes:

project.json

datasets

annotations

models

experiments

spatial assets

Export generates a manifest file for integrity checks:

{
  "files": [
    {"path": "datasets/images/img001.jpg", "hash": "abc123"},
    {"path": "models/onnx/model.onnx", "hash": "def456"}
  ],
  "project_version": "1.0.0"
}


Import verifies hashes to ensure no corruption.

Export is zip-based for portability, including cross-platform use.

5. Project Isolation

Projects are fully self-contained:

No shared datasets/models unless explicitly linked

Avoids accidental overwriting

Allows multiple projects per machine safely

Engines are aware of project paths, never write outside project folder by default.

6. Project Metadata & Engines Interaction

On project load:

UI reads project.json

Registers available engines and versions

Checks execution mode vs hardware

Engines read project paths, dataset registry, and model registry

Any outputs (inference, experiments, spatial reconstructions) are saved inside project folder for reproducibility.

7. Cross-Platform Storage Considerations

File paths stored as relative paths to project_root:

datasets/images/img001.jpg → relative

Prevents absolute path issues on Windows / macOS / Linux

Optional symbolic links for very large datasets:

e.g., if user wants to store videos externally, still register in project.

8. Security & Access Control

Project folder can be encrypted for sensitive industries (defense, health)

Execution mode can enforce:

Offline-only: block network calls

Hybrid: allow cloud engines for heavy tasks

Audit logs written per project:

Who ran inference

When experiments were trained

Remote vs local execution

9. Workspace UI Features (Electron)

Project dashboard shows:

Dataset summary (#images, #videos, geo coverage)

Quick actions:

10. Hybrid / Remote Mode Behavior

Projects can exist entirely locally, but optionally use:

Online 2D → 3D engine

Project = self-contained, lazy-loaded, versioned workspace

DATASET & DATA MANAGEMENT ENGINE — DETAILED TECHNICAL SPEC

Goal: Handle all types of datasets efficiently, support large images/videos, allow spatial/geospatial operations, version everything, and keep memory usage low.

Architecture Principle:

The Dataset Engine runs as a background Python service. The Electron frontend communicates via async IPC / gRPC / REST.
Heavy operations like tiling, parsing, and indexing are offloaded to worker threads or processes.

A. Supported Input Types & Technical Handling
Input Type	How Engine Handles It	Notes / Libraries
Images	Copied or referenced; stored in datasets/images/	Pillow, OpenCV
Video	Frames extracted on ingest or on-demand	OpenCV, FFmpeg; store frame metadata separately
Satellite Imagery	Large images automatically tiled	Rasterio, GDAL
GeoTIFF	Parsed and linked to spatial metadata	GDAL, Rasterio
CSV / Tabular Spatial Data	Parsed; stored in spatial index	Pandas, Shapely, GeoPandas
B. Data Ingestion Pipeline

Pipeline Overview:

Raw Input
  → Validation
  → Metadata Extraction
  → Optional Tiling
  → Indexing
  → Dataset Registration

1. Validation

Check supported file formats

Check for missing frames / corrupted images

Validate CSV / GeoTIFF headers

Generate ingestion log with warnings/errors

2. Metadata Extraction

For each asset, store:

{
  "id": "uuid",
  "file_path": "datasets/images/img001.jpg",
  "type": "image/video/frame",
  "size": [width, height],
  "duration": null,        # for video
  "frame_index": null,     # for video frames
  "timestamp": "...",
  "geo_bounds": {          # optional
      "lat_min": ...,
      "lat_max": ...,
      "lon_min": ...,
      "lon_max": ...
  },
  "source": "drone/satellite/user"
}


Metadata stored in SQLite / DuckDB

Enables fast querying without loading raw files

3. Optional Tiling (Critical for Large Images / Maps)

Images larger than threshold (e.g., 2048x2048) are split into tiles

Tile metadata keeps:

Original coordinates

Tile coordinates

Tile index

Example structure:

original_image_id: "img001"
tiles:
  - tile_1: {x0, y0, width, height, lat/lon bounds}
  - tile_2: ...


Tiling allows:

Large satellite images

Map overlays

Memory-efficient CV training

4. Indexing

All assets are registered in the Dataset Registry

Index contains:

file path

asset type (image/video/frame)

timestamp

spatial bounds (lat/lon)

class labels (if annotated)

Storage:

SQLite / DuckDB table dataset_index

Optional R-tree for geo-spatial queries

Example schema:

CREATE TABLE dataset_index (
    asset_id TEXT PRIMARY KEY,
    file_path TEXT,
    type TEXT,
    width INT,
    height INT,
    timestamp DATETIME,
    lat_min REAL,
    lat_max REAL,
    lon_min REAL,
    lon_max REAL,
    labels TEXT
);

C. Dataset Versioning & Snapshots

Purpose: Prevent accidental overwrites, allow experiments to reproduce results.

Ingest / Update Dataset

Compare with previous version → identify changes

Save diff record (added / removed / modified files)

Assign dataset version ID (v1.0, v1.1, uuid)

2. Snapshot Model

Each snapshot is a read-only reference:

Original files remain untouched

Diffs stored in JSON:

{
  "snapshot_id": "uuid",
  "timestamp": "...",
  "added": ["img005.jpg", "img006.jpg"],
  "removed": ["img003.jpg"],
  "modified": []
}


Snapshots allow rollback:

User can revert to any previous version

Required for experiments reproducibility

D. Lazy Loading / Memory Management

Only load:

Tile metadata / indices

Dataset statistics

Actual images/videos are read on demand

Enables handling:

Very large satellite datasets

High-res video sequences

Supports chunked processing for CV tasks

Video Ingestion Options:

H. Data Pipeline Integration

User Action → Dataset Engine → Vision Engine → Training / Inference → Outputs Engine


All CV tasks query dataset registry, not filesystem

Relative paths only

COMPUTER VISION TASKS — DETAILED TECHNICAL SPEC

Goal: Enable robust CV operations (classification, detection, segmentation, tracking, and change detection) across images, video, and spatial data. All tasks are engineered for modularity, reproducibility, and integration with other engines.

Architecture Principle:

The Vision Engine is a Python service handling training, inference, and evaluation.
Electron frontend interacts via async RPC / REST / IPC.
The engine can run locally (CPU/GPU) or remotely (online accelerator, e.g., Hugging Face / cloud).

A. Task Abstraction

All CV tasks implement a common interface, making pipelines consistent:

class VisionTask:
    def __init__(self, model_path: str, device: str = "cpu"):
        self.model_path = model_path
        self.device = device
        self.model = None
    
    def load_model(self):
        """
        Load ONNX / PyTorch / TensorFlow model.
        Handles device allocation (CPU/GPU).
        """
        pass

    def preprocess(self, data):
        """
        Convert input data to model-compatible tensors.
        Handles:
            - resizing / padding
            - normalization
            - tiling for large images
            - geospatial metadata if needed
        """
        pass

    def infer(self, preprocessed_data):
        """
        Run model inference.
        Supports:
            - batch inference
            - live video stream
            - spatially-referenced tiles
        """
        pass

    def postprocess(self, raw_outputs):
        """
        Convert raw model outputs to:
            - bounding boxes
            - segmentation masks
            - labels / confidence
            - spatial coordinates (geo-ref)
        """
        pass


Benefits:

Unified interface for all tasks

Swap models easily (custom / pretrained)

Supports ONNX standardization for cross-framework compatibility

Works with both local and online engines

B. Change Detection (Expanded)

Change detection is critical for mapping, defense, mining, and industrial monitoring.

1. Concept

Detect differences between two or more temporally-separated images, optionally with geo-awareness.

Inputs:

image_1, image_2 (or more)

Optional geo-metadata for alignment

Optional video sequence (frame-by-frame)

Outputs:

Change mask (binary or multi-class)

Area statistics (e.g., hectares of deforestation)

Geo-linked differences (GeoJSON / GeoTIFF overlays)

Confidence / probability per pixel (if learned)

2. Pipeline

Step 1: Preprocessing

Align images: Ensure images have same size and coordinate system.

Use image registration:

Keypoint-based (SIFT / ORB / SuperPoint)

Feature matching → affine/homography transform

Optional: geo-coordinates for geospatial alignment

Normalize: Resize / scale / normalize to model input

Tile large images: For high-res maps or satellite imagery

Step 2: Inference / Comparison
Two main strategies:

A. Pixel / Feature Difference

diff = abs(image_1 - image_2)
mask = diff > threshold


Simple, fast

Works for illumination-normalized images

Good for MVP / Level 1

B. Learned Change Detection Model

Input: stacked images ([img1, img2])

Architecture: CNN / UNet / Siamese network

Output: per-pixel probability map of change

Advantages:

Robust to illumination, weather, seasonal differences

Can classify type of change

Step 3: Postprocessing

Threshold probability map → binary mask

Optional morphological operations:

Remove noise

Close gaps

Calculate change statistics:

Area of change (pixels → real-world units using geo_ref)

Count of objects changed (if detection integrated)

Save geo-linked overlay:

GeoJSON polygons

GeoTIFF raster mask

Optional PLY for 3D environments

Goal: Optimize resource usage, reduce friction, and allow heavy workloads to be offloaded to cloud or hybrid execution, without breaking offline-first behavior.

A. Core Concept — Smart Task Orchestrator

Every heavy task in the app passes through a central orchestrator before execution.

Tasks include:

Model training

Inference on large datasets or videos

3D reconstruction (SfM / SLAM)

Change detection on high-resolution maps

Orchestrator decisions depend on:

Hardware capability

CPU cores

GPU presence & type (CUDA, OpenCL, Apple M1/M2)

RAM size

Disk space

Dataset size

Number of images

Resolution

Number of frames (for video)

3D point cloud / mesh size

User plan / license

Free / premium (some offloading may be premium-only)

User preferences

Force local / hybrid / remote

Security / offline mode

Decision matrix:

Dataset Size	GPU Available	Plan	Suggested Execution
Small	Yes / No	Any	Local
Medium	GPU yes	Free	Local
Medium	GPU no	Free	Local (CPU slow)
Large	Any	Premium	Hybrid / Remote
Extra Large	Any	Any	Remote
B. Technical Flow

1. Pre-Execution Assessment

Called before running any engine-intensive task.

Checks:

hardware_specs = detect_hardware()
dataset_stats = get_dataset_stats(dataset)
user_plan = get_user_plan()
security_settings = get_project_security()


Computes estimated runtime & memory requirement per engine:

Vision Engine → model size + batch size

3D Engine → number of frames + resolution

Spatial Engine → map size + tiles

If resource requirement > available, flag for offload.

2. Execution Mode Decision

Modes:

Local

All engines run on user's PC

Use GPU if available

CPU fallback if GPU unavailable

Offline-capable

Remote / Cloud

Heavy computation (3D reconstruction, large model training) runs on remote engine

User PC acts as client / orchestrator

Data optionally uploaded securely

Results streamed back asynchronously

Hybrid

Preprocessing & lightweight tasks run locally

Heavy tasks offloaded

Example:

Tile satellite imagery locally

Offload large-scale 3D reconstruction to remote engine

Download reconstructed mesh + CV overlays for local visualization

C. Local Resource Management

Engine throttling

Limit number of threads / GPU usage per task

Prevent app from freezing

Memory management

Lazy loading of datasets

Stream large images/videos instead of loading fully into RAM

Use memory-mapped files for GeoTIFF, point clouds

Task Queue

Background queue for long-running tasks

Prioritizes tasks based on user choice

Async & Parallel Execution

Use multiprocessing or asyncio in Python

For Electron frontend: use Web Workers to avoid blocking UI

D. Remote / Cloud Offloading

Optional engine (like your online 2D → 3D engine)

Execution flow:

Serialize task config (dataset pointers, model, params)

Securely upload to remote engine

Monitor progress via API

Stream results / download outputs once complete

Security features

Optional end-to-end encryption

Optional anonymization before upload (especially for Defense / Health)

Advantages

Reduces local memory / CPU/GPU load

Enables very high-resolution or large datasets

Supports premium / enterprise workflows

E. Hybrid Mode (Intelligent)

Combines local preprocessing + remote heavy computation

Example:

Local: Tiling satellite images, indexing datasets, running small model inferences

Remote: Full model training or 3D reconstruction

Results are merged automatically:

Local annotations aligned with remote 3D outputs

Overlays geo-referenced automatically

Workflow:

Local Task → Preprocess → Upload Subset → Remote Engine → Run Task → Return Results → Merge & Visualize Locally

F. Performance Optimization Techniques

Lazy Loading: Load datasets/models only when needed

Tile-based processing: Process large images or maps in chunks

GPU Acceleration: Use ONNX Runtime / TensorRT for inference

Batch Processing: Batch frames/images for efficient computation

Streaming for Video / 3D: Avoid holding full video or mesh in RAM

Profiling: Keep runtime logs for optimization (time per task, memory usage)

G. Developer Notes

Smart Task Orchestrator is a central service that wraps engine calls.

Must query project profile to respect constraints (e.g., Defense = no cloud).

Must estimate task resources before execution and decide execution mode.

Supports local / remote / hybrid modes dynamically.

Integrates with dataset engine, vision engine, 3D engine for precise load assessment.

Must provide feedback to UI:

ETA estimates

Task status

Warnings if local execution is slow

✅ Takeaway:

Performance & Offloading ensures Singularity Vision runs efficiently on small laptops without sacrificing advanced capabilities, by intelligently deciding between local, remote, or hybrid execution based on hardware, dataset size, user plan, and industry profile constraints.
